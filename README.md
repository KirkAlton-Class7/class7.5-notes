# Class 7.5 / SEIR-1 Notes 

## Why
If I am teaching I will typically use these notes to organize class structure, provide documentation, static assets, artifacts, scripts, etc. 

Sometimes I will take notes if I am not teaching for demonstration purposes; to prepare for other classes; if its new material; or some other similar reason. 

## Usage
Treat my notes as an _additional_ resource. Ideally you aren't looking at them during class but writing your own notes. 

You should be taking your own notes. This is a skill you need to develop not only to pass certifications or study but to bolster your technical writing skills. You need these documentation authoring skills as a critical "soft skill". On the job you will be expected to efficently and concisely document issues you encounter, expected behavior, observed behavior and possible solutions (to write tickets for example); summarize other documentation and resources you have used so that team members can quickly get the same knowledge (if you are tasked with researching a tool or testing a solution of some kind); to write out a method and strategy for lower level engineers to solve problems independently (a runbook); or many other things. 

## Organization

I will organize notes into weeks and if needed days. A week or day may contain a markdown file (readme.md) that is the core of the notes, additional markdown files may be provided, there may be an assets folder with diagrams, and perhaps a code folder with shell scripts, python or terraform. 

## Disclaimer 
I can only gurantee these notes accuratently reflect class if I teach the class. If another teacher is giving instruction then they might cite different documentation, use different diagrams, write code slightly different, etc. My notes should always be up to date and work correctly while arriving at the same solution but how I explain something, automate something, architect something or any other problem solving could be a bit different. In general it should be essentially the same. Any differences would be a great oppurtunity for you to ask why I did it one way as opposed to someone else doing it another way. Maybe there is a better solution and why is it different for example. 

Finally, Theo is the final say. If he wants something done a certain way then that should always be presumed to be the all instuctors or admins expect it to be done.
